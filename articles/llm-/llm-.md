> 🎯 **预防胜于治疗！在 50% 时处理，比在 95% 时抢救要容易得多**

## 😱 你的 AI 是否也这样"失忆"过？

**场景 1**：
```
你：刚才说的那个函数叫什么？
AI：我不太理解你的意思...
（其实不是 AI 笨，是上下文被截断了）
```

**场景 2**：
```
你：继续刚才的代码
```

```
Error: This model's maximum context length is 128000 tokens
```

---

## 🎯 为什么需要上下文管理？

### LLM 的致命限制

所有大语言模型都有**固定的上下文窗口**：

| 模型 | 上下文窗口 | 约等于 |
|------|-----------|--------|
| GPT-4 Turbo | 128K tokens | 10 万汉字 |
| Claude 3 | 200K tokens | 15 万汉字 |
| Qwen Max | 32K tokens | 2.5 万汉字 |

**但对话是无限增长的！**

### 撑爆后的三种灾难

| 灾难 | 症状 | 严重程度 |
|------|------|---------|
| 😱 **静默截断** | AI 偷偷丢掉最早的消息，你完全不知道 | ⭐⭐⭐⭐⭐ |
| 💥 **API 报错** | 返回 400：`context_length_exceeded` | ⭐⭐⭐⭐ |
| 🧠 **AI 失忆** | 忘记前面说过的话，回答质量断崖下降 | ⭐⭐⭐⭐⭐ |

**最可怕的是静默截断，因为你完全感知不到发生了什么。**

## 🚨 三级预警系统

我们在 OpenClaw 中实现了**运行时上下文监控**，在消息送入 LLM 之前就检查 token 使用量，提前预警。

### 预警级别

| 级别 | 阈值 | 状态 | 行动要求 |
|------|------|------|---------|
| 🟡 **黄色预警** | 50-75% | 需要注意 | 开始准备 |
| 🟠 **橙色预警** | 75-90% | 立即行动 | 压缩/归档 |
| 🔴 **红色告警** | 90-100% | 紧急情况 | 立即清空 |

### 预警卡片示例

```
┌────────────────────────────────────────────────┐
│ 🟠 上下文预警                                  │
├────────────────────────────────────────────────┤
│ 当前使用：78.3%      紧急程度：注意            │
│ Token 统计：已用 100,234 / 剩余 27,766         │
├────────────────────────────────────────────────┤
│ 💡 建议操作：                                  │
│ [🗜️ 压缩对话] [📦 归档会话] [🧹 清空上下文]   │
└────────────────────────────────────────────────┘
```

## 📋 分级应对策略（核心章节）

### 🟢 正常阶段（0-50%）

**状态**：健康，无需担心

**特征**：
- ✅ 对话刚刚开始
- ✅ AI 记忆完整
- ✅ 响应速度快

**建议操作**：
- ✅ 正常对话
- ✅ 无需任何操作
- ✅ 享受流畅的 AI 体验

**示例场景**：
```
你：帮我写一个 Python 脚本
AI：好的，请问具体需要什么功能？
你：需要爬取网页数据...
```

---

### 🟡 黄色预警阶段（50-75%）

**状态**：需要注意，开始准备

**特征**：
- ⚠️ 对话已经进行了较长时间
- ⚠️ AI 还能记住大部分内容
- ⚠️ 但需要开始考虑收尾

**建议操作**：

#### 1️⃣ 评估当前对话价值

问自己三个问题：
1. 这次对话还有继续的必要吗？
2. 前面的内容是否重要？
3. 是否需要保存历史记录？

#### 2️⃣ 选择应对策略

| 场景 | 建议操作 | 耗时 |
|------|---------|------|
| 对话快结束了 | 直接继续，不用管 | 0 秒 |
| 还需要继续聊 | 点击"压缩对话"按钮 | 1 秒 |
| 内容很重要 | 点击"归档会话"保存 | 3 秒 |
| 已经聊偏了 | 点击"清空上下文"重新开始 | 1 秒 |

#### 3️⃣ 压缩对话（推荐）

**操作**：
```bash
# 方法 1：点击卡片按钮
[🗜️ 压缩对话]

# 方法 2：发送命令
/summarize

# 方法 3：手动要求
AI：请总结前面的对话，保留核心内容
```

**压缩后效果**：
```
AI：✅ 对话已压缩到 45%

保留内容：
- 项目目标：开发博客系统
- 技术栈：Node.js + Express + MongoDB
- 已完成：用户认证、文章 CRUD
- 下一步：评论功能、标签系统

已删除：
- 重复讨论
- 试错过程
- 无关闲聊
```

#### 4️⃣ 实际案例

**场景**：你正在开发项目，已经聊了 50 分钟

**收到黄色预警后**：
```
你：/summarize

AI：✅ 对话已压缩

已保存核心内容：
- 项目目标：开发博客系统
- 技术栈：Node.js + Express + MongoDB
- 已完成：用户认证、文章 CRUD
- 待完成：评论功能、标签系统

当前上下文：45%（已释放 28% 空间）
```

**继续对话**：
```
你：好，接下来实现评论功能
AI：好的！评论功能需要支持：
    1. 用户发表评论
    2. 回复评论
    3. 点赞/点踩
    先从哪个开始？
```

### 🟠 橙色预警阶段（75-90%）

**状态**：⚠️ **需要立即行动**

**特征**：
- 🚨 上下文即将耗尽
- 🚨 AI 可能开始出现"失忆"症状
- 🚨 继续对话风险很高

**建议操作**：

#### 🚨 必须三选一

**选项 A：压缩对话（适合还要继续聊）**

```bash
# 方法 1：点击卡片按钮
[🗜️ 压缩对话]

# 方法 2：发送命令
/summarize

# 方法 3：手动要求
AI：请总结我们前面的对话，保留核心内容
```

**压缩后检查**：
```
AI：✅ 已压缩到 55%

保留内容：
- 项目架构设计
- 数据库模型
- API 接口定义

已删除：
- 重复讨论
- 试错过程
- 无关闲聊
```

---

**选项 B：归档会话（适合重要对话）**

```bash
# 方法 1：点击卡片按钮
[📦 归档会话]

# 方法 2：发送命令
/archive

# 方法 3：手动保存
AI：请把这次对话保存到 Obsidian
```

**归档后效果**：
```
📦 对话已归档

保存位置：
Obsidian/archive/2026-04-22-项目讨论.md

包含内容：
- 完整对话记录
- 代码片段
- 设计文档
- 决策记录

当前上下文已清空，可以开始新对话了！
```

---

**选项 C：清空上下文（适合重新开始）**

```bash
# 方法 1：点击卡片按钮
[🧹 清空上下文]

# 方法 2：发送命令
/clear

# 方法 3：手动要求
AI：我们重新开始，忘记前面的内容
```

**清空后效果**：
```
🧹 上下文已清空

已删除：
- 所有历史消息
- 临时变量
- 上下文依赖

保留：
- 系统提示词
- Bot 基础配置

现在可以开始全新的对话了！
```

---

#### ⚠️ 橙色预警时的禁忌

**❌ 不要做的事**：
1. **不要继续长对话** - 很容易撑爆
2. **不要发送大段代码** - 会加速消耗
3. **不要开启新话题** - 先处理当前上下文
4. **不要忽略警告** - 很快会到 90%

**✅ 应该做的事**：
1. **立即压缩或归档** - 释放空间
2. **评估对话价值** - 决定是否继续
3. **保存重要内容** - 避免丢失
4. **考虑分话题讨论** - 避免单一对话过长

### 🔴 红色告警阶段（90-100%）

**状态**：🚨 **紧急情况！必须立即处理！**

**特征**：
- 🔴 上下文即将耗尽
- 🔴 AI 随时可能"失忆"
- 🔴 下一条消息可能导致 API 报错

**建议操作**：

#### 🚨 紧急处理流程

**Step 1：立即停止发送新消息**

```
❌ 不要再发："帮我看看这个代码有什么问题..."
✅ 应该先处理上下文告警
```

**Step 2：快速选择处理方案**

| 你的需求 | 推荐操作 | 耗时 |
|---------|---------|------|
| 还要继续聊 | 点击"🧹 清空上下文" | 1 秒 |
| 内容很重要 | 点击"📦 归档会话" | 3 秒 |
| 想保存 + 继续 | 先归档，再清空 | 5 秒 |

**Step 3：验证处理结果**

```
处理前：
🔴 上下文告警：93%

处理后：
✅ 上下文已清空：5%（系统提示词）
```

---

#### 🚑 红色告警时的应急预案

**场景 1：正在讨论关键问题，突然红色告警**

```
你：等等，先别说了，上下文快满了！
    /archive  （先归档）
    /clear    （再清空）
    好了，继续刚才的话题...
```

**场景 2：红色告警时 AI 已经开始胡言乱语**

```
你：/clear  （立即清空）
    你刚才说的不对，我们重新来
    问题是：{重新描述问题}
```

**场景 3：红色告警 + 重要代码还没保存**

```
你：/archive  （先归档保存）
    代码我已经保存了
    我们继续...
```

---

#### ⚠️ 红色告警的严重后果

**如果不处理，会发生什么？**

1. **AI 失忆** - 忘记前面说过的话
   ```
   你：刚才说的那个函数叫什么？
   AI：我不太理解你的意思...
   （其实是被截断了）
   ```

2. **API 报错** - 直接返回错误
   ```
   Error: This model's maximum context length is 128000 tokens
   ```

3. **数据丢失** - 未保存的内容全部丢失
   ```
   你：等等，我还没保存代码！
   （但已经晚了，上下文被强制截断）
   ```

## 💀 撑爆后的恢复方案

**如果你看到了红色告警，但没有及时处理，然后...**

### 症状

- AI 开始胡言乱语
- 回答与问题无关
- 说"我不太理解"
- API 返回 400 错误

### 恢复步骤

#### Step 1：立即清空上下文

```bash
/clear
```

#### Step 2：尝试恢复对话

```
你：我们刚才在讨论 XXX 问题
    已经完成了 ABC
    接下来要做 XYZ
    请继续...
```

#### Step 3：从归档中恢复（如果有归档）

```bash
# 查看归档文件
cat Obsidian/archive/2026-04-22-*.md

# 复制关键内容
你：这是前面的讨论摘要：
    {粘贴归档内容}
    
    我们继续...
```

## 🎯 核心原则总结

### 三句话记住

1. 🟡 **黄色预警时开始准备** - 不要等到 75%
2. 🟠 **橙色预警时立即行动** - 不要拖到 90%
3. 🔴 **红色告警时果断清空** - 不要犹豫

### 三个投资

| 投资 | 回报 |
|------|------|
| 花 1 分钟压缩 | 节省 10 分钟恢复时间 |
| 花 3 分钟归档 | 避免重要数据丢失 |
| 花 5 分钟清空 | 避免 AI 失忆 |

### 三个禁忌

1. ❌ 不要忽略黄色预警
2. ❌ 不要拖延橙色预警处理
3. ❌ 不要在红色告警时继续发送消息

---

## 📊 对比：处理 vs 不处理

| 场景 | 处理方式 | 结果 |
|------|---------|------|
| 50% 时压缩 | 花 1 分钟 | ✅ 继续流畅对话 |
| 50% 时忽略 | 继续聊 | ⚠️ 很快到 75% |
| 75% 时压缩 | 花 2 分钟 | ✅ 释放 30% 空间 |
| 75% 时忽略 | 继续聊 | ⚠️ 很快到 90% |
| 90% 时清空 | 花 1 分钟 | ✅ 重新开始 |
| 90% 时忽略 | 继续发 | 💥 AI 失忆 + 数据丢失 |

**结论**：预防的成本远低于恢复的成本！
