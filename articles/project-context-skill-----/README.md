## 📋 摘要

本文记录了 Project Context Skill 从设计到实施的全过程，**重点对比了"代码固化"与"Skill 固化"两种方案的差异及选型依据**。

**核心问题**：在多 Agent 协作流程中，如何避免下游 Agent 重复理解上游已分析的内容？

**实施策略**：采用"Skill 工具化 + 文档约束"模式，而非代码级强制集成。

**关键决策依据**：
1. 模型能力判断：2026 年的模型已能准确遵循复杂文档
2. 流程稳定性：上下文传递仍在探索期，需要灵活性
3. 错误成本：最坏情况是重新理解问题，损失 5-10 分钟，可控
4. 长期维护：Skill 固化改文档即可，代码固化需要技术变更

**当前状态**：2026-04-23 实施完成，进入 2 天观察期。2026-04-25 将根据使用数据决定是否全面推广。

## 🎯 问题背景

### 场景：多 Agent 协作中的上下文传递

在 CI/CD 流水线中，多个 Agent 依次参与：

```
CTO 助理 → CTO → OpenCode → Code Reviewer → Deploy Assistant
```

**问题**：每个下游 Agent 都需要重新理解上游的分析结果。

**典型案例**（2026-04-22）：
1. Tech Assistant 花费 30 分钟分析"微信支付成功后页面不跳转"问题
2. TA 完成分析报告（包含问题定位、根因分析、修复方案）
3. CTO 助理将分析结果写入飞书任务看板
4. CTO 被唤醒后，**重新阅读 TA 报告，重新理解问题**（5-10 分钟）
5. OpenCode 被派工后，**再次阅读原文和 TA 报告**（5-10 分钟）

**效率损失**：
- CTO 重新理解问题：约 5-10 分钟
- OpenCode 重新理解问题：约 5-10 分钟
- 整体效率损失：约 67%（根据《上下文传递规范 v1.0》估算）

## 💡 设计思路：代码固化 vs Skill 固化

### 核心洞察

> "随着大模型越来越聪明，文档约束也应该越来越可行。"
> —— 用户，2026-04-23

这一洞察直接影响了我们的技术选型。在实施过程中，我们深入对比了三种方案。

---

### 方案 A：代码固化（Code-First）

#### 实现方式

```javascript
// CTO 助理主程序（cto-assistant.js）
async function dispatchToCTO() {
  // 强制：初始化上下文
  const ctx = new ProjectContext(projectPath);
  await ctx.init();
  
  // 强制：写入 TA 分析
  await ctx.write('taAnalysis', taAnalysis);
  
  // 强制：验证完整性
  const valid = await validateContext('cta');
  if (!valid) throw new Error('上下文不完整');
  
  // 强制：导出并传递
  const context = await ctx.export();
  await sessions_spawn({ agentId: 'cto', context });
}
```

#### 优点
- ✅ **100% 执行** - 流程写死在代码中，不会忘记
- ✅ **流程标准化** - 无歧义，所有 Agent 行为一致
- ✅ **错误即时捕获** - 编译时/运行时检查

#### 缺点
- ❌ **灵活性差** - 改流程需要改代码、测试、重启服务
- ❌ **维护成本高** - 每次调整都是技术变更
- ❌ **调试复杂** - 需要看代码才能理解流程
- ❌ **不信任模型** - 与"模型越来越聪明"的趋势相悖

#### 适用场景
- 模型能力弱（2024 年之前）
- 流程高度稳定，几乎不变
- 错误成本极高（如金融、医疗）

---

### 方案 B：Skill 固化（Skill-First）⭐ 最终选择

#### 实现方式

```bash
# CTO 助理 SOUL.md（文档描述）
## Project Context Skill 集成

每次派发任务前，必须执行：
1. node init-context.js <项目路径>
2. node write-context.js --type taAnalysis --data '...'
3. node validate-context.js --stage cto
4. node export-context.js
```

#### 优点
- ✅ **灵活性高** - 改文档即可调整流程
- ✅ **维护成本低** - 无需编译、测试、部署
- ✅ **易于理解** - 自然语言描述，人类和模型都能看懂
- ✅ **工具标准化** - 提供统一 Skill，避免重复实现
- ✅ **信任模型** - 给模型学习和遵循规范的空间

#### 缺点
- ⚠️ **依赖模型自觉** - 可能忘记使用
- ⚠️ **无编译时检查** - 错误在运行时才发现
- ⚠️ **需要验证机制** - 兜底检查是否遵守

#### 适用场景
- 模型能力强（2026 年及以后）
- 流程快速迭代，需要频繁调整
- 错误成本可控（如内部工具）

---

### 方案 C：纯文档约束（Doc-Only）

#### 实现方式

```markdown
# SOUL.md - CTO 助理行为规范

## 上下文传递

派发任务时，应该使用 ProjectContext 工具缓存分析结果，
避免下游 Agent 重复理解。具体用法参考相关文档。
```

#### 优点
- ✅ **最大灵活性** - 完全依赖模型理解
- ✅ **零技术成本** - 只需写文档

#### 缺点
- ❌ **完全依赖模型自觉**
- ❌ **无标准化工具** - 每个 Agent 自己实现
- ❌ **格式不统一** - 难以互操作

#### 适用场景
- 模型能力极强（未来）
- 流程极其简单
- 临时性、一次性任务

---

### 方案对比总结

| 维度 | 代码固化 | Skill 固化 ⭐ | 纯文档约束 |
|------|---------|------------|-----------|
| **执行力** | ✅ 100% | ⚠️ 依赖自觉 | ❌ 完全自觉 |
| **灵活性** | ❌ 低 | ✅ 高 | ✅✅ 极高 |
| **维护成本** | ❌ 高 | ✅ 低 | ✅✅ 极低 |
| **调试难度** | ❌ 复杂 | ✅ 中等 | ✅✅ 简单 |
| **工具标准化** | ✅ 统一 | ✅ 统一 | ❌ 各自实现 |
| **适用模型** | 弱模型 | 中等 - 强模型 | 极强模型 |

### 最终决策：Skill 固化

#### 决策依据

**1. 模型能力判断**
- 当前模型（2026 年）已能准确遵循复杂文档
- 趋势：模型越来越聪明
- 结论：可以信任模型自觉使用 Skill

**2. 流程稳定性**
- 上下文传递流程仍在探索期
- 可能需要频繁调整
- 结论：需要灵活性，避免代码固化

**3. 错误成本**
- 最坏情况：下游 Agent 重新理解问题
- 损失：5-10 分钟时间
- 结论：错误成本可控，可接受一定失败率

**4. 长期维护**
- 代码固化：每次调整需要技术变更
- Skill 固化：改文档即可
- 结论：Skill 固化更适合快速迭代

#### 最终决策

> 采用 **Skill 固化（Skill-First）** 模式，提供标准化工具 + 文档约束，而非代码级强制集成。

#### 风险缓解

1. **提供标准化工具**（降低使用门槛）
2. **设置验证机制**（兜底检查）
3. **建立监控系统**（及时发现问题）
4. **定期评估**（2 天后决策是否调整）

#### 升级路径

```
Skill 固化（当前）
    ↓
观察 2 天，收集数据
    ↓
使用率 > 80% → 继续 Skill 固化，推广到其他 Agent
使用率 50-80% → 优化 Skill 接口 + 增加提醒
使用率 < 50% → 升级为代码固化（关键路径）
```

## 🤔 反思与教训

### 做得好的

1. **洞察准确**："文档约束优先"基于对趋势的判断
2. **工具先行**：先提供标准化工具，降低使用门槛
3. **监控完备**：实施同时创建监控系统
4. **渐进策略**：先观察再决策，风险可控

### 潜在风险

1. **依赖模型自觉**：如 CTO 助理忘记使用，Skill 形同虚设
2. **项目命名约束**：假设项目不重名
3. **性能开销**：每次调用启动 Node.js 进程（约 0.1-0.2 秒）

### 可改进的

1. **自动化程度**：可在 sessions_spawn 时自动注入 context
2. **错误提示**：如忘记使用 Skill，可更明确提示
3. **性能优化**：减少进程启动开销
