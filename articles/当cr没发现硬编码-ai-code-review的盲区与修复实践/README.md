# 当CR没发现硬编码——AI Code Review的盲区与修复实践

> **摘要**：一段代码中硬编码了与配置文件冲突的值，资深架构师级别的 Code Reviewer 却没能发现。这不是CR的失职，而是LLM审查的结构性盲区。本文剖析根因、给出修复方案，并探讨如何让AI Code Review真正可靠。

---

## 📋 前言

在最近的代码评审中，发生了一个值得深思的事件：

> 一段代码中硬编码了一个配置值，而这个值在配置文件中已经有明确定义，甚至与配置文件中的值**直接冲突**。代码的硬编码不仅架空了配置文件，还导致系统在运行时使用了错误的配置。

更令人意外的是——**Code Reviewer（CR）没有发现这个问题**。

CR的人设是"资深架构师级别的评审专家"，拥有完整的代码审查checklist。但即便在这样的条件下，硬编码问题依然溜过了审查。

这引出了一个核心问题：**AI Code Review的盲区在哪里？我们该如何修复？**

---

## 🎯 一、问题重现

### 1.1 场景还原

```python
# config.yaml
database:
  max_connections: 50
  timeout: 30
```

```python
# database.py
class Database:
    def __init__(self):
        # 硬编码了 max_connections，与配置文件冲突
        self.max_connections = 100  # ❌ 应该从配置读取
        self.timeout = 30           # ❌ 同样硬编码
        self.connect()
```

配置文件定义 `max_connections: 50`，但代码中硬编码为 `100`。系统运行时实际使用了 `100`，配置文件被完全架空。

### 1.2 CR的检查结果

CR的审查报告如下：

```
✅ 代码逻辑正确，功能符合预期
✅ 命名规范，可读性良好
✅ 异常处理完备
✅ 无安全漏洞
⚠️ 建议：考虑添加单元测试
```

**CR没有标记硬编码问题**。

---

## 🤔 二、根因分析

为什么"资深架构师级别"的CR会漏掉这个明显的问题？

### 2.1 Checklist 的盲区

CR的checklist包含以下条目：

- 代码逻辑正确性
- 命名规范与可读性
- 异常处理
- 安全漏洞
- 性能影响
- 单元测试覆盖

**但没有"硬编码 vs 配置文件一致性"这一项**。

这是第一个根因：**CR的审查维度是功能导向的，不是结构一致性导向的**。

### 2.2 LLM审查的天然局限

LLM作为Code Reviewer，其审查模式存在结构性局限：

| 维度 | LLM 擅长 | LLM 盲区 |
|------|---------|---------|
| **单文件审查** | ✅ 代码逻辑、命名、异常处理 | — |
| **跨文件一致性** | ❌ 难以关联多个文件的语义 | 硬编码 vs 配置 |
| **上下文广度** | ✅ 当前文件完整上下文 | ❌ 项目级全局状态 |
| **约束检查** | ✅ 语法、类型、规范 | ❌ 架构约束、配置一致性 |

LLM天然关注**当前文件的内部一致性**，而**跨文件的结构一致性**需要额外的机制来保障。

### 2.3 人格设定 ≠ 执行能力

CR被设定为"资深架构师级别的评审专家"，但这只是一个**人格设定（persona）**。

人格设定告诉LLM"你是谁"，但**执行需要落地为具体的动作**。如果没有明确的检查指令，LLM不会自动执行架构师级别的跨文件审查。

> **关键洞察**：人格设定是理想，具体指令是现实。没有落地的指令，再高的人格设定也无法转化为实际能力。

## 🛠️ 三、修复方案

### 3.1 在AGENTS.md中新增"配置一致性检查"

我们在 AGENTS.md 的硬编码检测板块新增了 **P0级** 的"配置一致性检查"：

```markdown
### 🔴 P0：配置一致性检查（新增）

**触发条件**：代码中出现硬编码值（数字、字符串、URL、路径等）

**检查步骤**：
1. 读取项目配置文件（config.yaml / .env / settings.py 等）
2. 建立期望配置项列表
3. 逐项核对代码中的硬编码值是否与配置一致
4. 发现冲突 → 标记为 P0 问题，要求修复
```

### 3.2 明确跨文件校验步骤

修复后的检查流程：

```
┌─────────────────────────────────────────────┐
│           跨文件配置一致性检查                │
├─────────────────────────────────────────────┤
│  Step 1: 扫描代码中的硬编码值                 │
│          ↓                                  │
│  Step 2: 定位相关配置文件                    │
│          ↓                                  │
│  Step 3: 逐项比对硬编码值与配置值             │
│          ↓                                  │
│  Step 4: 发现冲突 → P0 问题                  │
│          ↓                                  │
│  Step 5: 生成修复建议                        │
└─────────────────────────────────────────────┘
```

### 3.3 代码层面的防御性编程

除了CR层面的修复，代码层面也应引入防御性机制：

```python
# ✅ 推荐做法：强制从配置读取
from config import Config

class Database:
    def __init__(self):
        self.max_connections = Config.database.max_connections
        self.timeout = Config.database.timeout
        self.connect()

# ❌ 避免做法：硬编码
class Database:
    def __init__(self):
        self.max_connections = 100  # 硬编码
```

修复后，CR的审查报告变为：

```
✅ 代码逻辑正确，功能符合预期

✅ 无安全漏洞

🔴 P0：发现硬编码冲突 — database.py:12 max_connections=100 
       与 config.yaml:5 max_connections=50 不一致，请修复
```

### 4.2 系统自检机制

我们在 AGENTS.md 中建立了三层硬编码检测体系：

| 层级 | 检测方式 | 覆盖范围 |
|------|---------|---------|
| **L1** | 代码静态扫描 | 单文件硬编码识别 |
| **L2** | 跨文件配置比对 | 硬编码 vs 配置文件冲突 |
| **L3** | 运行时校验 | 配置加载时的值一致性检查 |

## 🤔 五、反思与教训

### 5.1 人格设定不能替代具体指令

这次事件最深刻的教训是：**再高的人格设定，如果没有落地的具体指令，也无法转化为实际能力**。

CR被设定为"资深架构师"，但架构师的审查能力不是自动获得的，而是需要通过：

1. **明确的检查清单**（checklist）
2. **具体的执行步骤**（step-by-step）
3. **可验证的输出标准**（output criteria）

来实现的。

### 5.2 LLM的审查需要"脚手架"

LLM不是全能的。它的审查能力需要**脚手架（scaffolding）**来支撑：

- 脚手架 = 检查清单 + 执行步骤 + 输出标准
- 没有脚手架，LLM只能做它"本能"会做的事
- 有脚手架，LLM可以执行超出其本能范围的任务

### 5.3 从错误中学习，让错误永不重演

Harness Engineering 的核心理念是：

> *"Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again."*
> — Mitchell Hashimoto

这次硬编码漏检事件，我们通过以下方式确保它**永不重演**：

1. ✅ AGENTS.md 新增 P0 级配置一致性检查
2. ✅ 建立三层硬编码检测体系
3. ✅ 在CR工作流中强制跨文件校验步骤
4. ✅ 代码层面推广防御性编程实践

配置一致性只是跨文件一致性的一种。未来可扩展到：

- **API接口一致性**：代码中的API调用与API文档的一致性
- **数据库Schema一致性**：代码中的模型定义与数据库Schema的一致性
- **依赖版本一致性**：代码中的依赖引用与依赖管理文件的一致性
